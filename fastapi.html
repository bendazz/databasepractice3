Question 1 code from memory

Assume the imports and the app instance are already in place:

from fastapi import FastAPI

app = FastAPI()

Write a single GET endpoint that returns the string "Hello".

Answer
@app.get("/hello")
def hello():
    return "Hello"

Reasoning. An endpoint in FastAPI has two parts:

  • A decorator @app.get(...) that binds the function below it to an HTTP method and URL path. Here GET /hello.
  • A function whose return value becomes the response body. Returning the Python string "Hello" produces the JSON response "Hello".
Question 2 short answer

Your app is running and its base URL is homepage. You want to test the endpoint from the previous question:

@app.get("/hello")
def hello():
    return "Hello"

What URL would you put in the browser's address bar to hit this endpoint directly?

Answer

homepage/hello

Reasoning. The full URL for a FastAPI endpoint is the app's base URL followed by the path given to the decorator. The decorator was @app.get("/hello"), so the path is /hello. Appending that to the base URL homepage gives homepage/hello. Typing that in the browser issues a GET request — the method the endpoint is registered for — and the browser displays the returned string.

Question 3 code from memory

Assume the imports and the app instance are already in place:

from fastapi import FastAPI

app = FastAPI()

Write a GET endpoint that takes in an integer and returns its square.

Answer
@app.get("/square")
def square(n: int):
    return n * n

Reasoning. In FastAPI, any function parameter that does not appear in the decorator's path is treated as a query parameter. Since {n} is not in the path /square, n becomes a query parameter. The : int annotation tells FastAPI to parse the value from the URL (a string) into an int, and to reject the request with a validation error if it isn't a valid integer.

Question 4 short answer

Your app is running and its base URL is homepage. You want to test the endpoint from the previous question with n = 5. What URL would you put in the browser's address bar?

Answer

homepage/square?n=5

Reasoning. Query parameters come after the path, separated by a ?, and each one is written as name=value. The endpoint's path is /square and the query parameter is n, so passing 5 means appending ?n=5. The full URL is the base URL plus the path plus the query string: homepage/square?n=5.

If there were more than one query parameter, they would be joined with & — e.g., ?n=5&m=3.

Question 5 code from memory

Assume the imports and the app instance are already in place:

from fastapi import FastAPI

app = FastAPI()

Write a GET endpoint that takes in two integers and returns their sum.

Answer
@app.get("/sum")
def add(a: int, b: int):
    return a + b

Reasoning. Neither a nor b appears in the decorator's path, so both are query parameters. Each : int annotation tells FastAPI to parse that value from the URL as an integer and to reject the request with a validation error if it isn't one. The function returns a + b, which FastAPI serializes as JSON.

Question 6 short answer

Your app is running and its base URL is homepage. You want to test the endpoint from the previous question with a = 7 and b = 4. What URL would you put in the browser's address bar?

Answer

homepage/sum?a=7&b=4

Reasoning. The query string starts with ?, and multiple query parameters are joined with &. The endpoint's path is /sum and its parameters are a and b, so appending ?a=7&b=4 to the base URL plus path gives homepage/sum?a=7&b=4. The order of the parameters in the query string doesn't matter — ?b=4&a=7 would work equally well.

Question 7 matching

In a FastAPI + SQLModel app, Inventory is a table model and engine is its database connection. Below are four endpoint function bodies (the decorators have been removed). Match each function to the English description of what it does.

Function 1

def operation_1(item: Inventory):
    with Session(engine) as session:
        session.add(item)
        session.commit()
        session.refresh(item)
        return item

Function 2

def operation_2(id: int):
    with Session(engine) as session:
        statement = select(Inventory).where(Inventory.id == id)
        record = session.exec(statement).one()
        session.delete(record)
        session.commit()
        return {"deleted": id}

Function 3

def operation_3():
    with Session(engine) as session:
        statement = select(Inventory)
        items = session.exec(statement).all()
        return items

Function 4

def operation_4(id: int, item: Inventory):
    with Session(engine) as session:
        statement = select(Inventory).where(Inventory.id == id)
        record = session.exec(statement).one()
        record.name = item.name
        record.quantity = item.quantity
        record.price = item.price
        session.add(record)
        session.commit()
        session.refresh(record)
        return record

Descriptions:

  1. Remove a single row from the Inventory table, looked up by its id.
  2. Return every row currently stored in the Inventory table.
  3. Insert a new row into the Inventory table.
  4. Find a row by its id and overwrite its fields with new values.
Answer
  • Function 1 → C (insert a new row)
  • Function 2 → A (remove a row by id)
  • Function 3 → B (return every row)
  • Function 4 → D (find by id and overwrite fields)

Reasoning. Read each body for the telltale SQLModel calls:

  • Function 1 receives a whole Inventory object, calls session.add(item) followed by commit, and returns the refreshed object. add + commit with no prior lookup means insert.
  • Function 2 looks up a single row with select(...).where(Inventory.id == id), then calls session.delete(record) and commit. That's a delete.
  • Function 3 does select(Inventory) with no where clause and uses .all() to return every matching row. With no filter, that's read all.
  • Function 4 looks up a row by id, then reassigns its attributes from the incoming item before committing. That's an update.

The decorators for these in the warehouse app were @app.post("/inventory"), @app.delete("/inventory/{id}"), @app.get("/inventory"), and @app.put("/inventory/{id}") respectively — but you can tell what each one does just by reading the body.
